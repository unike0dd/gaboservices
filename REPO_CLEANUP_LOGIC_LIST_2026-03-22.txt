======================================================================
2. IMPROVE / CONSOLIDATE CANDIDATES
======================================================================

A. CONSOLIDATE DUPLICATED FORM LOGIC

Files:
- contact/contact-hub.js
- careers/careers-form.js
Problem:
- The two files carry overlapping helper logic for selection pills, state resets, validation flow, and submit-status handling.
Logic:
- Duplicate business logic increases drift, creates double maintenance, and raises the chance that one form gets fixes the other one misses.
Improvement:
- Extract one shared configurable form engine or helper module, then keep only page-specific field rules in the page-level scripts.
Expected result:
- Fewer bugs, easier testing, and faster future changes across both form experiences.

B. CONSOLIDATE DUPLICATED METADATA NORMALIZATION

Files:
- main.js
- site-governance.js
Problem:
- Both files rebuild or merge site metadata from the same sources.
Logic:
- Metadata normalization should have one canonical owner; otherwise defaults, fallbacks, and merge rules can diverge over time.
Improvement:
- Move the metadata merge into one shared helper and import it where needed.
Expected result:
- Single-source-of-truth metadata behavior.

C. SIMPLIFY FAB / MOBILE-NAV INITIALIZATION OWNERSHIP

Files:
- main.js
- chatbot/chatbot-controls.js
- fab-controls.js
Problem:
- initFabControls() can be invoked through more than one path.
Logic:
- Even if the current dataset guard prevents visible breakage, double-init patterns are unnecessary complexity and can cause future regressions.
Improvement:
- Choose one module as the only owner of FAB/mobile-nav startup and let all other modules depend on that ownership model.
Expected result:
- Clearer startup order and less side-effect coupling.

D. MAKE SERVICES DATA EITHER FUNCTIONAL OR SMALLER

File: services/index.html
Problem:
- Service objects include href values, but the current rendering path does not appear to use those URLs.
Logic:
- Inert properties make the data model misleading. They suggest an interaction model that the page does not actually provide.
Improvement:
- Either wire href into actual service-card navigation or CTA behavior, or remove the property from the data structure.
Expected result:
- Cleaner content model with no misleading fields.

E. SIMPLIFY LOW-VALUE CHATBOT GUARD / PROBE LOGIC

File: chatbot/chatbot-controls.js
Problems:
- ChatbotTinyGuard appears to validate site-controlled trigger text or attributes rather than defending against a clearly separate threat path.
- Gateway availability probing appears to degrade into a permissive path for cross-origin endpoints, which weakens the value of the probe.
Logic:
- If a protective layer does not materially change risk or behavior, it should be simplified, documented, or removed.
Improvement:
- Decide whether the trigger guard solves a real abuse case.
- Remove it if it does not.
- Replace pseudo-health checks with a documented real probe strategy or a simpler fail-open UX.
Expected result:
- Chatbot controls that are easier to reason about and maintain.

F. IMPROVE DISCOVERABILITY WHERE THE PAGE EXISTS BUT IS WEAKLY SURFACED

File: learning/index.html
Problem:
- The page is available and appears in supporting navigation or sitemap paths, but it is not strongly surfaced in primary navigation.
Logic:
- If Learning is strategic, weak discoverability is a product problem; if it is not strategic, the current page may not justify its maintenance cost.
Improvement:
- Make an explicit decision:
  * promote it into primary navigation and cross-linking, or
  * keep it intentionally secondary, or
  * remove or archive it if it is not serving a product purpose.
Expected result:
- The page's visibility matches its actual business importance.
